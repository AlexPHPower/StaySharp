# StaySharp

> Keep your coding reflexes sharp through deliberate, repeatable practice.

StaySharp is a developer kata tool powered by Claude. It generates coding challenges — functions with intentional bugs hidden across realistic multi-file structures — and paired test suites. You find the bug, fix it, verify with tests, then archive your results and get a fresh batch. No two runs are the same.

---

## Why StaySharp?

Most kata tools give you isolated puzzles — a single function, a single input/output pair, no surrounding context. Real bugs don't work like that. They hide inside helper functions, validators, formatters, and calculation utilities that the main function delegates to. You have to read the code, understand what it's supposed to do, trace the logic across multiple files, and figure out where the contract breaks.

StaySharp generates challenges that reflect this. A challenge might be a discount calculator that calls a tax utility and a currency formatter — the bug is in one of those three files, and you won't know which until you investigate. The test suite tells you something is wrong; it's your job to find it.

The tool is also regenerative. Once you've solved a batch, `/run-suite` archives your results and generates a completely fresh set. You can run this daily, weekly, or whenever you want to stay sharp.

---

## How It Works

StaySharp has no challenge files committed to the repo. Everything is generated by Claude at runtime using two slash commands:

| Command | What it does |
|---|---|
| `/create-suite` | Generates a batch of challenge directories + test files |
| `/check` | Runs all test suites and reports progress — nothing is changed |
| `/run-suite` | Runs tests, archives results, regenerates fresh challenges |
| `/setup` | Installs all language dependencies from the repo root |

The typical loop:

```
/create-suite → fix challenges → /check → fix more → /run-suite → repeat
```

## Quickstart

### 1. Prerequisites

You need the following runtimes installed:

| Language | Required |
|---|---|
| Python | Python 3.11+ |
| TypeScript | Node.js 18+ |
| Go | Go 1.23+ |
| Java | JDK 21+, Maven 3.9+ |

You don't need all four. StaySharp works with any subset — just specify `languages=` when generating challenges.

### 2. Install dependencies

Open this repo in Claude Code and run:

```
/setup
```

This installs pytest, vitest, runs `go mod tidy`, and downloads Maven dependencies — all from the repo root. It's safe to run multiple times and skips anything already installed.

Or manually:

```bash
pip install pytest
cd typescript && npm install
cd go && go mod tidy
cd java && mvn install -DskipTests
```

### 3. Generate your first challenge batch

```
/create-suite
```

Generates 5 challenges distributed across all four languages by default. Each challenge gets its own subdirectory and may span multiple files.

To customise:

```
/create-suite count=3 languages=python,typescript
```

### 4. Solve the challenges

Open the challenge directories and read the code. Each challenge has:

- A docstring at the top of the main file describing what the function is supposed to do
- A `# BUG:` comment on the defective line naming the problem type (not the fix)
- Supporting files that may or may not contain the bug

Your job: figure out what's wrong and fix it.

### 5. Check your progress

At any point, run `/check` to see the current state across all active challenges:

```
/check
```

This runs all test suites and reports pass/fail per challenge. Nothing is archived or deleted — it's a read-only progress report. Use it as often as you like between fixes.

You can also run tests directly for a single language:

### 5a. Run tests manually per language

```bash
# Python
cd python && python -m pytest tests/ -v --tb=short

# TypeScript
cd typescript && npm test

# Go
cd go && go test ./challenges/... -v

# Java
cd java && mvn test -q
```

A passing test suite means your fix is correct. A failing test means either the fix is wrong or you haven't found the right bug yet.

### 6. Archive and regenerate

Once you're done with a batch (or want to move on):

```
/run-suite
```

This runs all active test suites, writes a results file to `results/YYYY-MM-DD_HH-MM-SS.md`, prints a summary, then deletes the challenge files and generates a fresh batch automatically.

## Challenge Structure

### Subdirectory per challenge

Each challenge lives in its own subdirectory. A challenge spans 1–4 files depending on difficulty:

```
python/challenges/discount_calculator/
    __init__.py
    discount_calculator.py      ← main entry-point
    tax_calculator.py           ← supporting utility
    currency_formatter.py       ← another supporting file

python/tests/
    test_discount_calculator.py ← imports from the challenge directory
```

Supporting files are real, working code — they are not decoys. The bug is hidden in exactly one file, and it could be anywhere in the chain.

### Difficulty and complexity

| Difficulty | Typical files | Bug location |
|---|---|---|
| Easy | 1 | Main file |
| Medium | 2–3 | Main or one helper |
| Hard | 3–4 | Buried in a helper |

### What the `# BUG:` comment tells you

The comment names the category of defect — not the fix:

```python
# BUG: off-by-one
if index < len(items) - 1:
```

This tells you what kind of error to look for. It does not tell you what the correct code should be.

## Challenge Categories

Each run draws from these categories. Maximum one per category per language per run — so you always get variety:

| Category | Examples |
|---|---|
| Algorithm | Binary search, sorting, graph traversal, recursion |
| String manipulation | Parsing, formatting, pattern matching, encoding |
| Business logic | Discount tiers, eligibility rules, approval workflows |
| Financial calculations | Interest, tax, currency rounding, fee structures |
| Data validation | Input sanitisation, format checks, constraint enforcement |
| Date/time business logic | Business days, deadlines, scheduling, timezone handling |
| Data transformation | Aggregation, filtering, reshaping, normalisation |

---

## Defect Types

Each challenge contains exactly one intentional defect. Defect types are varied across each run:

| Type | Description | Example |
|---|---|---|
| Logic bug | Wrong operator or condition | `and` instead of `or` |
| Off-by-one | Boundary wrong by exactly one | `<` instead of `<=` |
| Wrong formula | Correct-looking math with wrong constant or operation | Divides by 100 instead of 1000 |
| Incomplete implementation | Returns early or skips a case entirely | Missing branch for zero/negative input |
| Inverted condition | Condition is negated when it shouldn't be | `not is_valid` when `is_valid` is the correct guard |

## Slash Commands Reference

### `/setup`

```
/setup
```

Checks and installs all language dependencies from the repo root. Prints a status summary for each language. Run this once after cloning, or any time a language environment is broken.

---

### `/create-suite [count=N] [languages=...]`

```
/create-suite
/create-suite count=3
/create-suite languages=python,go
/create-suite count=2 languages=typescript,java
```

Generates `count` challenges distributed evenly across the specified languages. For each challenge, Claude generates all files (main, supporting, test) simultaneously.

After generating, Claude prints a summary table:

```
| # | Language   | Challenge            | Files                                    | Category     | Difficulty | Defect            | Bug location      |
|---|------------|----------------------|------------------------------------------|--------------|------------|-------------------|-------------------|
| 1 | Python     | discount_calculator  | discount_calculator.py, tax_calculator.py| Financial    | medium     | wrong formula     | tax_calculator.py |
| 2 | TypeScript | parseUserInput       | parseUserInput.ts, validators.ts         | Validation   | easy       | inverted condition| validators.ts     |
```

Claude will not overwrite existing challenge directories. If challenges already exist, it skips them and notes it in the summary.

---

### `/check`

```
/check
```

Runs all active test suites and reports results per challenge. **Nothing is modified** — no files are archived, deleted, or regenerated.

Use this as often as you like mid-session. Example output:

```

──────────────────────────────────────────────
Python
  ✅  discount_calculator   4/4 tests passing
  ❌  date_range_validator  2/3 tests passing
        FAILED: test_excludes_weekends

TypeScript
  ✅  parseUserInput        3/3 tests passing

Go
  ❌  inventory_counter     1/4 tests passing
        FAILED: TestHandlesZeroStock
        FAILED: TestNegativeQuantityReturnsError
        FAILED: TestTotalWithMixedItems

──────────────────────────────────────────────
Overall: 10/14 passing across 4 challenges
When you're ready to finish: /run-suite
```

---

### `/run-suite`

```
/run-suite
```

No arguments. Detects which languages have active challenge files, runs their test suites, archives results, and regenerates.

**Step by step:**

1. Detects active languages (those with non-`.gitkeep` files in challenge directories)
2. Runs each language's test suite — never aborts if one fails, collects all results
3. Parses total/passed/failed counts and failing test names
4. Reads challenge metadata (category, difficulty) before deleting anything
5. Writes `results/YYYY-MM-DD_HH-MM-SS.md`
6. Prints a terminal summary:
   ```
   StaySharp Run — 2025-06-14 09:15:00
   ─────────────────────────────────────
   Python      ✅  4/4 passed
   TypeScript  ❌  2/3 passed  (1 failed)
   Go          ✅  3/3 passed
   ─────────────────────────────────────
   Results saved to: results/2025-06-14_09-15-00.md
   ```
7. Deletes challenge directories and test files (`.gitkeep` files preserved)
8. Calls `/create-suite` with the same count and languages to regenerate

## Tips

**Start with one language.** If you're new to StaySharp, run `/create-suite count=3 languages=python` to get a small focused batch before going multi-language.

**Read before you grep.** The challenge docstring tells you what the function should do. Understand the intent first — then read the implementation and look for where it diverges.

**Let the tests guide you.** If you're stuck, run the tests and read the failure output carefully. The failing assertion often points to the boundary or case the bug affects.

**Don't look at the `# BUG:` comment first.** Try to find the bug without it. The comment is a hint, not a starting point — use it if you're well and truly stuck.

**Archive even partial attempts.** `/run-suite` archives whatever state the tests are in. If you fixed 3 of 5 challenges, that still gets recorded. You can see your improvement over time in the results files.
